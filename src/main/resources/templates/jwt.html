JWT Or Jason Web Token

  • Imagine a Hotel Room that you have booked for a 4 days vacation
  • When you check in, the Hotel Front Desk will make a couple of Hotel Room Access Card for you to enter your room
  • After you check out these key cards will be deactivated
  • So in real life, we do support / are aware of short living access cards / access keys compared to the Room Lock Keys that remains valid for as long as the lock is not changed
  • Why the Access Card is deactivated? Because the Hotel does not want unauthorized people to enter the rooms
  • Your Hotel Room is a physical property / resourced owned by the Hotel
  • Similarly, we have digital properties such as a customer, a product an order a task an item etc. that we serve from our REST APIs.
  • We also issue a short living access token for these API clients (like short duration hotel guests)
  • These tokens can be of various types like Jason Web token or Opaque Token etc
  • A JWT or Jason Web Token is an base 64 encrypted String with hree sub parts 1. The Header 2. The Payload 3. The signature
  • The subparts are separated with a DOT (.)
  • The Authorization Server encrypts such tokens with its private key issues these to the Client asking for them
  • The Client then presents these tokens to the REST API (Resource Server) as an Authorization Header
  • The REST API / Resource Server has the public key of the server key pair and can validate the access token
  • Once the REST API / Resource Server validates the access token (JWT) it serves the API response.
