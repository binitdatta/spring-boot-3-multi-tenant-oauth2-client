Welcome To Interface Driven Programming

  • Imagine a Retail Bank and the Teller Counters it has
  • The actual Bank Employees who sit behind the Teller Counters keep changing but the Teller counters work the same way for the customers
  • Thus, the Teller Counters implement a uniform Interface and all the employees who sit (aka implements) the Teller Counter, works the same way
  • Now imagine driving 5 different cars all of which has the Gas and Break pedals
  • Irrespective of the Make and Model and Year, all Gas and Break Pedals work / behave the same way
  • This Car Manufacturing is another example of Interface Driven Manufacturing keeping the outward behavior same with inner differences hidden
  • Information Technology industry and Java, Spring heavily draw upon this Interface Driven Concept
  • In the Java Enterprise Edition (JEE) all the functionality provided under different modules like JDBC, ORM (JPA) , JMS and others are just INTERFACES nothing else
  • These interfaces (JDBC/JPA) for example are implmented by providers like Spring, Hibernate, EclipseLink, Oracle, MySQL, MSFT SqlServer, IBM DB2 and so on, ActiveMQ, RabbitMQ, IBM MQ etc
  • In Spring MVC and Spring Security itself we will encounter many many such interfaces and the implementations will either be written by Spring or by us
  • In this Application, the CustomerClient is an interface telling Spring where the Server lives (host and port) and what URL it has to call in what HTTP Method
  • At Runtime Spring Boot writes a dynamic implementation (if you debug you can see the class) and it works
