Spring Boot (3) to Boost Productivity

  • Servlet Filters first came into life during the first release of Java Enterprise Edition in 1996/1997.
  • We can imagine Servlet Filters like the US Transportation Security Administration (TSA) in US Airports.
  • Like TSA can stop, check and allow / deny entry all travellers in airports, Servlet Filters do the same for Web Traffic
  • Spring Security heavily uses Servlet Filters right from the first version and even till now.
  • In TSA Security scanning, some travellers with prior clearance (PreCheck) get to pass through
  • In Servlet Filter driven Spring Security also, we can identify Web URLs for public static assets and URLs like the landing page, the login page to be cleared without Security scanning
  • Much of these Spring Security Configuration was XML based earlier and now it is Java DSL Based
  • First, For any URLs, that Spring Security was not told to let go, it will stop the request with its Authentication Filter (1)
  • Second, the Spring Security Authentication Filter would extract the username and password (if any) and store them in a specialized object called UsernamePasswordAuthenticationToken
  • Third, Spring Security Authentication Filter would receive the UsernamePasswordAuthenticationToken
  • Fourth, Spring Security Authentication Filter would then delegate to the Spring Security AuthenticationManager for authenticating the user
  • Fifth, The Spring Security AuthenticationProvider is a concrete class that implements the AuthenticationManager. The ProviderManager can have a list of AuthenticationProviders all of which impelments the AuthenticationManager
  • Sixth, The Spring Security AuthenticationProvider invokes another interface driven (UserDetailsService) Spring Security Class
  • As we can see, The UserDetailsService can be implemented by a variety of InMemory, JDBC, LDAP and other classes (all from Spring Security) all of which return a standard interface Object UserDetails
  • We can also provide our own custom implementation of the Spring Security UserDetailsService interface when our user database tables do not match the structure Spring Security wants.
  • Seventh, The Implementation of UserDetailsService returns an object of UserDetails Interface to the AuthenticationProvider
  • Eighth, the AuthenticationProvider returns the UserDetail object to the AuthenticationManager
  • Ninth, the AuthenticationManager returns the UserDetail object to the AuthenticationFilter
  • Tenth, the AuthenticationFilter returns the UserDetail object to the SecurityContext
  • For subsequent requests by the same now logged-in user, Spring Security can retrive the token / cookies / JSESSION Cookie from the HttpServletRequest and check / match its presence in the SecurityContext, preventing further login by the same user.
