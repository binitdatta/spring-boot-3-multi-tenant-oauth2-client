Text Search With Elasticsearch

  • The best way to understand new technology and terms is to think ourselves as users.
  • Imagine you are driving through a US Freeway in a remote area , and you want to break for lunch or have some strong coffee
  • You stop in a rest area, and you google the nearest McDonalds, Subway, Starbucks, Dunkins or your favorite restaurant in your Mobile Phone
  • Millions of people across America do such searches and the responses needs to be superfast, like lightning fast
  • This use case is called Location Search and is used by all national companies / restaurant chains, pharmacy chains, and all businesses
  • Before such searches can be served, we need to understand the nature of the data and nature of existing storage platforms
  • RRelational Databases (RDBMS) are excellent storage platforms but they are not very good for taking in millions of requests and respond quickly.
  • Because Relational Databases support transactional integrity or ACID principles (Atomic, Consistent, Durable and Isolated)
  • All distributed storage platforms that can scale storage and retrieval as data volume / request volume grows, gives up ACID for that easy growth/scalability
  • Thus Elasticsearch is one such DISTRIBUTED (multiple computers or VMs work together to form a single team i.e. Cluster to store and serve data)
  • In Elasticsearch we have various types of players called Master Nodes, Data Nodes and Query Nodes
  • The Master Node accepts all connections from clients and routes them to either data nodes or query nodes depending on usecases
  • Master Node serve as a Router a traffic control / routing agent for the other data and query nodes
  • Data Nodes store data, replicate data stored in other data nodes for safety if the original crashes etc.
  • Query nodes parse Elastic Queries which are submitted through REST API and determines exact data locations, get data from those data nodes and combine the data and serve to the Clients
  • Elastic stores data in files called indexes but these indexes are divided into smaller segmented files called sherds in various data nodes
  • Each sherd data is also kept into two other data nodes apart from the original (normally) to prepare for node loss.
  • Like in RDBMS, one table stores multiple Rows, Elastic Index stores multiple Document, one Document is comparable to one RDBMS Row and one Table is comparable to one Index
  • Like in RDBMS, one row stores multiple columns, Elastic Document stores multiple Fields, one Field is comparable to one RDBMS Column
  • Now imagine when you are searching for a McDonald's location using your zipcode or Latitude / Longitude , Elastic is searching several Data Nodes IN PARALLEL because data is stored in distributed manner
  • Now imagine when you are searching for a McDonald's location using your zipcode or Latitude / Longitude , Elastic is searching several Data Nodes IN PARALLEL because data is stored in distributed manner
  • This parallel nature of Elasticsearch gives it the superfast response time that we see.
  • The other massive Elastic use case is product search in any huge Ecommerce platform or Supply Chain Platform that deals with millions (265 Million ) or Billions of Sellable Products online.
  • You can imagine the massive storage indexes for such a huge product index to support customer search fast
