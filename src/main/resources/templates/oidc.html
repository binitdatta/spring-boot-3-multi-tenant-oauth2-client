OIDC Stands for Open ID Connectivity

  • The first word is Open which means anyone can support it free of cost
  • Like there is an unspoken specification and industry agreement that all cars have four wheels.
  • When a new entrant Tesla starts making new Cars, it adheres to that agreement
  • Thus we can say the unspoken and largely agreed upon Car specification is open for all.
  • Similarly but more formally, the Internet Engineering Task Force (IETF) with the participation of all large IT organizations have created the OAuth Specification for Application Security
  • IETF released a first version in December 2007 called OAuth 1.0
  • When the first version was found to have several gaps, a second version called OAuth 2.0 was released on January 2013.
  • OAuth2 has formalized how clients and the services they call play their role along with the humans/machines they interact with
  • In OAuth2, there are four roles for each type of players/actors
  • First, Resource Owner is the Human User like us owning our Google / iCloud Photos
  • Second, Resource Server is the API (REST) serving the Resources (Data / Photos/ Documents that belongs to the user (Google Photos API)
  • Third, Client is the Application (Running on a Web / Mobile Browser or on a Mobile Tablet (or Capsule) Devices, making the data / photo / document requests
  • Fourth, as these sensitive data / photo / document requests cannot be served without proper verification, the Authorization Server roles authorizes the Clients on behalf of the user to get those data / photo / documents
  • Before OAuth, Organizations such as Home Mortgage Lenders asked for our bank account credentials to verify our income
  • Spreading our username and password to multiple organizations is a security hole. These companies can lose our credentials and when we change them in one place they get invalid
  • The most secured protocols of OAuth2 thus departs from having to ask user credentials and store them permanently.
  • OAuth2 Authorization Server ask the user to first sign in and then if they want to authorize the client application (for a short time) to access the data / photo etc
  • When the user consents to the data / photo access request from the Clients, the Authorization server stores these consents for the client applications that has their unique Client IDs
  • In subsequent times, when the client applications requests for an access token (typically valid for 15-30 minutes), the Auth. Server verifies the user consent and issues access tokens to the clients
  • The Clients then make data / photo / document requests through the Resource Server APIs and the Resource Server can validate the access tokens before serving the requests.
