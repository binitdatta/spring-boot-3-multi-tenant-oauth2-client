Aspect Oriented Programming (AOP)

  • The best way to understand AOP is to understand crosscutting concerns / Non Functional Requirements (NFR) in real Life
  • Lets imagine all the major Airlines, operating from the US, (Delta, United, British, Lufthansa, Qatar, Emirates and so on)
  • When the passengers of these massive airlines fly from a major airport, they and their luggage need security screenings
  • In the US , the Transportation Security Administration or TSA does that screening common to all Airlines, big or small
  • That means TSA or Security Screening is a common Aspect (English Word) of all Airlines in aviation. A lot of things in IT carry the same English meaning from their real life usage.
  • Now let us think about APIs or UIs that need Logging, Security, Auditing, Performance Metrics Collection and other Non-Functional Requirements
  • Doing these NFRs in 100s of APIs and UIs would be asking United and Delta to do their own security screening
  • That is when the Software Community learned Aspect Oriented Programming and implemented AOP in many frameworks like Spring Boot using various tools
  • But what are the tools used in AOP like TSA Agents are human beings that use xray machines, scanners, metal detectors etc?
  • Well, In AOP, the first thing that is used are Annotations
  • Annotations help identify which method of which class of which package needs to be served by AOP
  • Lets look at this : @Before("execution(public * com.rollingstone.controller.*Controller.*(..))")
  • We are saying that for any public method returning anything that belongs to a class ending in Controller and starting with anything else in a package com.rollingstone.controller and accepting any parameters will be a candidate for the method below
  • public void generalAllMethodASpect() { logger.info("All Method Calls invoke this general aspect method");}
  • A possible method candidate that can be used for AOP is called a JointPoint
  • A possible method candidate that has been used for AOP is called a PointCut
  • Annotations help us reduce redundant painful repetitions of code but the JVM does not understand them
  • Spring or any other framework has to actually modify the bytecode of the classes we write to make AOP work
  • One way is to use Java Reflection at runtime if you know what that is but Reflection is slow because of runtime class introspection
  • Another way is to use Java Proxy that intercepts all calls to the AOP impacted methods (we identified them using annotations above) and implement the requirements very much like TSA security for all Airlines
  • Another way is to use Bytecode instrumentation in which Spring modifies the actual code of our candidate classes at build/startuptime to do the heavy lifting on our behalf.
  • If we understand the above, then the roof and basement of AOP would be both clear to us.,.
